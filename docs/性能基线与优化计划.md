# 性能基线与优化计划

本文是 `CHARTDB-P6-000` 的执行清单，用于约束 Phase 6 的 bundle、Monaco、worker 和 large schema 性能优化。Phase 6 只处理本地浏览器性能，不引入账号登录、云端 diagram 存储、团队 Workspace、生产数据库直连或远端 schema 上传。

## 目标

- 降低首次进入编辑器时的首屏 JavaScript 体积。
- 避免 Monaco、模板数据、SQL/DBML parser 和自动 layout 在非必要路径阻塞主线程。
- 建立可重复记录的性能基线，让后续每个切片都能对比 build 输出和大 schema 行为。
- 在不支持 worker 或动态 import 失败时保留可理解的降级路径。

## 基线指标

| 指标 | 当前记录方式 | Phase 6 目标 | 后续任务 |
| --- | --- | --- | --- |
| 首屏 JS bundle | `npm run build` 输出的入口 chunk 和 gzip 后大小 | 主入口不主动包含 Monaco 和完整模板数据 | `CHARTDB-P6-001`、`CHARTDB-P6-002` |
| Monaco chunk | `npm run build` 输出中 Monaco/editor 相关 chunk | 仅 SQL、DBML、Smart Query 等代码编辑器路径加载 | `CHARTDB-P6-001` |
| 模板数据 chunk | `npm run build` 输出中 templates/examples 相关 chunk | 模板列表只加载 manifest，详情页再加载完整 diagram | `CHARTDB-P6-002` |
| large schema import 时间 | 浏览器 performance mark 或测试 helper 记录 100/500 tables import | 100 tables 不明显卡顿，500 tables 不冻结浏览器 | `CHARTDB-P6-003` |
| layout 主线程占用 | 导入后自动布局的耗时和 UI 可交互性 | 大 schema layout 迁移到 worker，主线程显示 progress/cancel | `CHARTDB-P6-003` |
| CI 性能预算 | build artifact 或脚本记录 bundle 变化 | 每次性能切片记录 chunk delta，Phase 7 再决定是否 CI 强制预算 | `CHARTDB-P6-004` |

## 执行顺序

1. `CHARTDB-P6-001`：Monaco 懒加载。
   - 先记录当前 `npm run build` chunk 输出。
   - 将 Monaco editor 从首屏路径拆出，只在 SQL、DBML、Smart Query 和代码展示需要时加载。
   - 验证代码编辑器路径仍可打开，记录 build chunk 变化。

2. `CHARTDB-P6-002`：模板 lazy registry。
   - 建立只包含名称、数据库类型、标签、描述和预览信息的 manifest。
   - 模板列表和示例列表不一次性 import 完整大型 schema。
   - 模板详情页按需加载完整 diagram，并保留错误提示。

3. `CHARTDB-P6-003`：Parser 和 layout worker 化。
   - 先把 large schema SQL/DBML import 的 parse 阶段迁移到 worker。
   - 再评估 layout 计算是否迁移到独立 worker。
   - worker error 必须回传结构化错误，UI 显示 progress 和 cancel。
   - 不支持 worker 的环境回退到同步路径，但要提示 large schema 可能较慢。

4. `CHARTDB-P6-004`：Bundle budget 和性能回归记录。
   - 汇总 Phase 6 的 build chunk delta。
   - 定义首屏 JS、Monaco chunk 和 templates chunk 的预算建议。
   - 决定哪些预算进入 CI，哪些只作为阶段验收记录。

## 测量方法

- 构建基线：运行 `npm run build`，记录入口 chunk、Monaco chunk、templates chunk 和警告。
- 大 schema 基线：使用 100 tables 与 500 tables 的 SQL/DBML fixture，记录 import parse、preview summary 和布局耗时。
- 交互基线：桌面和移动 smoke 至少覆盖首次进入、打开 Smart Query、打开 SQL/DBML editor、打开模板列表和导入 preview。
- 失败基线：动态 import 或 worker 初始化失败时，必须显示可理解错误，不能静默丢失 diagram。

## 验收门槛

- 文档中明确记录 bundle、Monaco、worker、large schema 和首屏指标。
- 后续代码切片必须在 run log 和阶段验收记录中写入 build chunk 变化。
- 性能优化不得牺牲 Phase 1 安全边界：不新增持久化 secret，不上传 schema，不绕过 import preview。
- 性能优化不得牺牲 Phase 5 可访问性：progress、error、cancel 和 disabled reason 必须有可访问名称或可读文本。

## 当前结论

`CHARTDB-P6-000` 只定义性能基线和执行顺序，不改变运行时代码。下一项应进入 `CHARTDB-P6-001`，优先处理 Monaco 懒加载并记录首屏和 Monaco chunk 变化。
