# 可选账号登录与团队协作预研

> 任务：`CHARTDB-P8-000`、`CHARTDB-P8-001`
> 定位：Phase 8 只做 Cloud/Team 产品边界和接口预研，不新增登录代码。

## 1. 结论

ChartDB 的默认产品形态仍是 OSS Core 本地优先模式：不登录仍可完整使用 OSS Core，用户无需账号、无需数据库密码，也不需要连接任何 ChartDB 服务端。

可选 Cloud/Team 应作为独立增强产品线推进。它可以提供账号登录、Workspace、Team、云同步、分享权限、版本历史和审计能力，但不能成为本地模式的前置依赖，也不能改变现有导入、编辑、导出、备份和恢复流程。

本轮只定义边界，不实现登录、OAuth、SSO、Workspace API、云端 diagram 存储、实时协作、评论、计费或权限代码。

## 2. OSS Core 保持不变

硬约束：

- 不登录仍可完整使用 OSS Core。
- 本地 diagram 不自动上传。
- 用户主动开启云同步后，才允许把指定 diagram 写入云端。
- 本地历史 diagram 不因登录而自动迁移到云端。
- AI Disabled、BYOK Session 和 Self-hosted Gateway 的安全边界不因 Cloud/Team 改变。
- 本地备份文件、DBML、SQL、PNG、SVG 和 Markdown 导出能力不依赖账号。

本地模式数据边界：

- 默认数据源仍是浏览器 IndexedDB。
- Smart Query 仍由用户在自己的数据库环境执行。
- ChartDB 前端不直接连接生产数据库。
- 导入内容、diagram 内容、note 和导出结果默认只在浏览器内处理。

## 3. 可选 Cloud/Team 能力范围

Cloud/Team 可以预留以下能力，但必须独立于 OSS Core 发布：

- 账号登录：Email/password、OAuth、SSO 只作为后续方案候选。
- Workspace：组织团队空间、成员列表和默认策略。
- Team：按团队组织 diagram、模板和分享链接。
- Diagram 云同步：用户显式选择同步目标，支持本地副本和云端副本分离标识。
- 分享权限：只读、可编辑、可复制三类最小权限。
- 版本历史：记录 diagram 版本、修改来源和恢复点。
- 审计日志：记录登录、成员变更、权限变更、分享链接创建、云同步、导出和删除。
- 数据治理：云端数据导出、数据删除、租户删除和保留期策略。

暂不纳入：

- 实时多人协作。
- 在线评论和审批流。
- 计费。
- 企业策略中心。
- 生产数据库直连。

## 4. 数据与接口边界

Cloud/Team 不应复用本地 IndexedDB 作为权威状态。建议后续单独定义服务端 Domain Model：

- `Account`：登录主体，不进入 OSS Core bundle 的默认运行路径。
- `Workspace`：租户级边界。
- `Team`：Workspace 内的协作组。
- `CloudDiagram`：云端 diagram 元数据和当前版本指针。
- `CloudDiagramVersion`：不可变版本快照。
- `ShareLink`：分享链接、权限和过期时间。
- `AuditLog`：服务端 AuditLog，记录 actor、target、action、timestamp、requestId 和 metadata。

接口 contract 应满足：

- 所有写入 API 必须带租户上下文。
- 所有 diagram 读取和写入必须校验 Workspace / Team 权限。
- 云同步 API 必须区分 create、pull、push、fork 和 disconnect。
- 删除 API 必须区分软删除、恢复、永久删除和数据导出前置检查。
- 导出 API 必须支持用户下载自己的云端数据。

## 5. 权限和审计策略

最小权限模型：

- Owner：管理 Workspace、成员、Team、分享链接、删除和导出。
- Maintainer：管理 Team diagram、版本和分享链接。
- Editor：编辑指定 diagram。
- Viewer：只读查看指定 diagram。

权限 negative tests 必须覆盖：

- 非成员不能读取 Workspace diagram。
- Viewer 不能编辑、删除、创建分享链接或导出私有 Workspace 数据。
- Editor 不能修改 Team 成员或 Workspace 设置。
- 被移除成员不能继续通过旧会话访问云端 diagram。
- 过期分享链接不能访问 diagram。
- 跨租户 ID 枚举不能读写其它租户数据。
- 断开云同步后，本地 diagram 不再向云端写入。

AuditLog 必须覆盖：

- 登录和登出。
- 成员邀请、接受、移除和角色变更。
- diagram 创建、同步、读取、导出、删除和恢复。
- 分享链接创建、权限变更、过期和撤销。
- 权限拒绝事件和跨租户访问尝试。

## 6. 隐私与合规边界

Cloud/Team 一旦启用，会把用户 schema、note、diagram layout 和版本历史保存到服务端，因此必须在产品和 API 上提供显式确认。

必须提供：

- 启用云同步前的确认说明。
- diagram 级同步开关。
- 云端数据导出。
- 云端数据删除。
- Workspace 删除和保留期说明。
- 敏感字段脱敏建议。
- 企业 SSO 或审计导出只作为后续可选项。

不得提供：

- 静默上传本地 diagram。
- 登录后自动同步全部历史 diagram。
- 把 API key、数据库密码或 Smart Query 连接凭据写入云端。
- 在 OSS Core 默认构建中硬依赖 Cloud/Team SDK。

## 7. Cloud/Team 技术栈选型记录

### 7.1 推荐方案

推荐方案：`NestJS + PostgreSQL + Auth.js/OIDC adapter + REST API + append-only sync log`。

选择原因：

- NestJS 更适合 Cloud/Team 的模块边界：Account、Workspace、Team、CloudDiagram、ShareLink 和 AuditLog 可以拆成独立 module，并通过 guard/interceptor 统一注入租户上下文和权限校验。
- PostgreSQL 适合 diagram metadata、版本快照、AuditLog、软删除和租户隔离；首轮可采用 shared database + tenant_id + row-level policy 设计，后续再评估 dedicated schema 或 dedicated database。
- Auth.js 适合先接 Email magic link、OAuth 和 OIDC；企业 SSO/SAML 不进入首轮实现，只保留 adapter 边界。
- REST API 更容易和 OSS Core 的本地优先 UI 分离；GraphQL/RPC 可作为后续内部管理端或批量查询优化选项，不作为首轮默认 contract。
- sync 采用显式 push/pull 和 append-only operation log，不做实时多人协作；conflict 首轮按版本号、baseVersion 和人工 merge 提示处理，不做 CRDT。

Cloud/Team 不作为默认依赖：以上依赖只允许进入独立 Cloud/Team package、deployment 或 self-hosted bundle，不能进入 OSS Core 默认构建，也不能让本地 IndexedDB、导入、导出、备份恢复依赖服务端。

OSS Core 隔离：

- OSS Core 继续以浏览器 IndexedDB 为权威状态，Cloud/Team 只通过用户主动启用的 sync adapter 读取或写入指定 diagram。
- 默认页面、模板、Smart Query、DBML/SQL import/export、backup/restore 不 import Cloud SDK。
- Cloud/Team API client 必须延迟加载，并且只在用户进入云同步设置或打开云端 Workspace 时初始化。
- 登录成功不触发自动上传；每个 diagram 需要显式绑定 cloudDiagramId 后才进入 sync。

### 7.2 方案 A：NestJS 托管 Cloud

组成：

- 后端：NestJS，按 Account、Workspace、Team、Diagram、Sync、AuditLog 拆 module。
- API：REST JSON，所有写入 API 统一校验 workspaceId、teamId、actorId 和 requestId。
- 数据库：PostgreSQL，核心表包含 account、workspace、team、membership、cloud_diagram、cloud_diagram_version、share_link、audit_log、sync_operation。
- Auth：Auth.js 或 OIDC provider adapter；OAuth 作为首轮候选，SAML/SSO 只定义接口边界。
- sync：显式 push/pull，服务端保存 baseVersion、headVersion 和 operation log；conflict 返回 `409 CONFLICT` 和可读 merge hint。

优点：

- 类型和模块边界清晰，权限 guard、AuditLog interceptor 和租户校验容易复用。
- 适合后续 Self-hosted Cloud 和商业 Cloud 共用服务端代码。
- PostgreSQL 事务能力可以保护版本写入、权限变更和 AuditLog 同步落库。

代价：

- 比单体轻量 API 初始复杂度高。
- Auth.js/OIDC、PostgreSQL migration、RLS 或 tenant_id guard 都需要独立安全测试。
- 需要额外部署边界，不能混入 OSS Core 静态部署。

### 7.3 方案 B：Fastify 轻量 API

组成：

- 后端：Fastify + TypeBox 或 Zod schema。
- API：REST JSON，路由按 workspace、diagram、sync、share、audit 分组。
- 数据库：PostgreSQL，保留和方案 A 相同的租户、版本和 AuditLog 模型。
- Auth：外部 OAuth/OIDC gateway 或轻量 JWT session，SSO 只做反向代理或企业版 adapter 预留。
- sync：同样采用 push/pull + operation log；conflict 返回结构化错误，不实现实时协作。

优点：

- 启动成本低，运行时轻，适合先做 self-hosted spike。
- Schema-first route validation 容易导出 API contract。
- 代码量少，适合只验证同步和权限模型。

代价：

- 模块、DI、guard 和 cross-cutting AuditLog 约束需要自己组织，长期演进风险高于 NestJS。
- 团队/权限/审计逻辑变多后，容易形成散落 route middleware。
- 企业 SSO、多租户策略和复杂审计导出需要补更多框架层约束。

### 7.4 暂不推荐方案：纯 BaaS 默认依赖

不推荐把 Supabase、Firebase 或其它 BaaS SDK 直接作为 OSS Core 默认依赖。

原因：

- 容易让本地模式被远端 SDK、登录初始化或服务端配置影响。
- 数据驻留、租户隔离、AuditLog 完整性和企业 SSO 边界会被平台能力约束。
- 若需要 BaaS，只能作为 Cloud/Team 独立 adapter，在 OSS Core 外部加载。

### 7.5 后续 API contract 草案

首轮只定义 contract，不实现：

- `POST /api/workspaces`：创建 Workspace。
- `GET /api/workspaces/:workspaceId/diagrams`：列出云端 diagram。
- `POST /api/workspaces/:workspaceId/diagrams`：创建 cloud diagram shell。
- `POST /api/diagrams/:cloudDiagramId/sync/push`：按 baseVersion 推送本地变更。
- `POST /api/diagrams/:cloudDiagramId/sync/pull`：拉取指定版本后的变更。
- `POST /api/diagrams/:cloudDiagramId/share-links`：创建分享链接。
- `GET /api/audit-log`：按租户、actor、target 和时间过滤审计事件。

每个接口都必须包含权限 negative tests，并验证跨租户 ID 枚举、旧 session、过期分享链接和低权限角色无法读写。

## 8. 后续技术预研问题

`CHARTDB-P8-001` 需要继续回答：

- 是否需要把 Auth.js/OIDC adapter 抽成独立 package。
- PostgreSQL tenant_id guard 和 row-level policy 是否同时启用。
- Cloud/Team self-hosted edition 是否允许关闭外部 OAuth，只使用本地 admin bootstrap。
- sync conflict 的人工 merge UI 是否进入下一轮产品计划。
- GraphQL 是否只用于管理端，还是继续坚持 REST-only Cloud API。

## 9. 验收

- 明确不登录仍可完整使用 OSS Core。
- 明确本地 diagram 不自动上传，云同步必须用户主动开启。
- 明确 Workspace、Team、权限、AuditLog、租户隔离、数据导出、数据删除和 negative tests。
- 明确本轮不新增登录代码，不改变 OSS Core 默认本地优先模式。
- 明确至少两种技术方案取舍：方案 A NestJS 托管 Cloud，方案 B Fastify 轻量 API。
- 明确推荐方案、OSS Core 隔离方式、Cloud/Team 不作为默认依赖、sync 和 conflict 策略。
