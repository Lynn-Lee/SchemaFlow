# SchemaFlow 发布检查清单

> 适用范围：SchemaFlow OSS Core 的 tag release、Docker image、自托管部署和维护发布。
> 默认边界：不要求账号登录，不要求数据库密码，不要求 AI key；AI 相关能力默认关闭。

## 1. 发布原则

- release 必须从 `main` 上的已验证提交产生，不从未合并分支直接发布。
- 发布前不得跳过 `test`、`build`、production `audit` 和 Docker smoke。
- 发布失败时先停止后续步骤，记录失败门禁、命令输出和修复负责人，不继续推送镜像或 release notes。
- 自托管 Docker image 不应把 OpenAI key、Gateway token、用户 schema 或本地 backup 写入镜像、bundle、日志或 `/config.js`。
- 任何 Cloud/Team、账号登录、Workspace、权限、计费和企业 SSO 能力都不属于 Phase 7 发布范围。

## 2. Release Gate

发布前必须确认：

- `git status --short --branch` 干净，当前提交已在 `origin/main`。
- `npm ci` 或等价干净安装成功。
- `npm audit --omit=dev --audit-level=high` 通过，production high/critical advisory 为 0。
- `npm run lint` 通过。
- `npm run test:ci` 通过。
- `npm run build` 通过，并记录主要 chunk 变化；既有大 chunk 警告必须关联到性能治理记录。
- `git diff --check` 通过。
- 安全扫描不新增浏览器端密钥、raw HTML Markdown 或危险 HTML 注入风险：

```bash
rg -n "VITE_OPENAI_API_KEY|OPENAI_API_KEY|window\\.env|rehype-raw|dangerouslySetInnerHTML" src Dockerfile default.conf.template .github
```

允许的已知命中必须说明原因，例如非敏感 `window.env` runtime object。

## 3. Docker Smoke

发布镜像前至少执行：

```bash
docker build -t schemaflow-release-smoke .
docker run --rm -d -p 8080:80 --name schemaflow-release-smoke schemaflow-release-smoke
curl -I http://localhost:8080/
curl -I http://localhost:8080/config.js
docker rm -f schemaflow-release-smoke
```

验收要求：

- 容器可以启动并返回静态页面。
- `/config.js` 返回 `Cache-Control: no-store`。
- 页面响应包含 CSP、`X-Content-Type-Options`、`Referrer-Policy`、`X-Frame-Options` 和 `Permissions-Policy`。
- 未配置 AI Gateway 时，应用仍可使用本地导入、编辑、导出和 backup/restore。

如果 Docker daemon 不可用，发布不得自动放行；必须在 run log 或 release checklist 中记录替代验证和人工补跑要求。

## 4. 文档 Gate

发布前检查文档是否覆盖真实能力：

- README 明确默认无账号、无数据库密码。
- `docs/部署与安全配置.md` 说明自托管安全头和 AI Gateway 边界。
- `docs/备份恢复格式.md` 说明 backup schema、版本和恢复规则。
- `docs/方言能力矩阵.md` 说明每个方言的 supported、partial、experimental 和 unsupported 范围。
- `docs/安全模型与AI边界.md` 说明 Disabled、BYOK Session 和 Self-hosted Gateway。
- release notes 不承诺未实现的 Cloud/Team、在线权限、实时协作或生产数据库直连。

## 5. 回滚与失败处理

- 如果 tag 已创建但 Docker smoke 失败，停止发布镜像并记录失败 tag、commit、失败命令和修复计划。
- 如果镜像已推送但发现安全或启动问题，发布补丁镜像并在 release notes 中标记受影响版本。
- 如果 production audit 出现 high/critical advisory，优先修复依赖；无法安全升级时阻断发布并登记风险。
- 如果文档和实际能力不一致，先修正文档或实现差异，不用 release notes 掩盖缺口。
