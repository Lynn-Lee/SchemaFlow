# 备份恢复格式

## 1. 格式目标

ChartDB backup 文件用于本地导出和恢复 diagram。Phase 3 首个版本只覆盖 OSS Core 的本地文件备份，不引入账号、云端存储、团队协作或远端同步。

## 2. 当前版本

当前格式：

```ts
type ChartDBBackupV1 = {
    format: 'chartdb.backup';
    schemaVersion: 1;
    createdAt: string;
    source: 'chartdb-local';
    appVersion?: string;
    diagramCount: number;
    diagrams: Diagram[];
};
```

字段约束：

- `format` 固定为 `chartdb.backup`，用于和旧的单 diagram JSON 区分。
- `schemaVersion` 固定为 `1`，backup 文件格式版本与 IndexedDB Dexie version 分开管理。
- `createdAt` 使用 ISO 8601 字符串，表示 backup 文件生成时间。
- `source` 固定为 `chartdb-local`，表示数据来自本地浏览器模式。
- `appVersion` 可选，用于记录导出时的 ChartDB 应用版本。
- `diagramCount` 必须等于 `diagrams.length`。
- `diagrams` 当前支持一个或多个完整 diagram payload；现有菜单导出入口先写入当前单个 diagram。

## 3. 导出规则

- 导出时必须写入 `format`、`schemaVersion`、`createdAt`、`source`、`diagramCount` 和 `diagrams`。
- 导出的 diagram 会重新生成内部 id，避免用户把同一 backup 多次导入时产生主键冲突。
- backup JSON 不包含账号、权限、云端 workspace、评论或远端同步信息。

## 4. 恢复规则

- 恢复前必须先解析 JSON，再校验 `format`、`schemaVersion`、`createdAt`、`diagramCount` 和 diagram 基础字段。
- `schemaVersion` 不是当前支持版本时，必须给出可理解错误，不写入 IndexedDB。
- `diagramCount` 与 `diagrams.length` 不一致时，必须拒绝恢复。
- 恢复为新 diagram 时必须重新生成 diagram id 和子实体 id，并刷新 `createdAt` / `updatedAt`。
- 旧版单 diagram JSON 暂时保留兼容导入路径，但新版导出统一使用 `chartdb.backup` 格式。

## 5. 后续扩展

- 多 diagram 选择和恢复摘要 UI 可在 Phase 5 体验优化中补齐。
- backup migration 与 IndexedDB migration 分开演进；新增版本时应在 `src/storage/backup` 中添加迁移和回归测试。
